# Worker 数据传输优化 - 性能对比报告

## 优化概述

**优化内容**：分离卫星元数据与位置数据，减少 Worker 到主线程的重复数据传输

**优化时间**：2026-04-08

---

## 测试环境

| 配置项 | 数值 |
|--------|------|
| 卫星数量 | 16,204 颗 |
| 更新间隔 | 3 秒 |
| 测试时长 | 15 秒（5 个更新周期） |
| 测试工具 | Chrome DevTools Performance |

---

## 性能对比汇总

| 指标 | 优化前 | 优化后 | 变化 |
|------|--------|--------|------|
| **Worker 每次传输量** | 2.16 MB | 0.85 MB | **-60.6%** |
| **每分钟传输总量** | 43.2 MB | 17.0 MB | **-60.6%** |
| **内存使用（稳定后）** | 192.3 MB | 163.7 MB | **-14.9%** |
| **平均 FPS** | 58.2 fps | 57.2 fps | -1.7% |
| **LCP** | 1,137 ms | 1,144 ms | +0.6% |
| **TLE API 耗时** | 1,471 ms | 1,296 ms | **-11.9%** |

---

## 详细数据

### 1. Worker 数据传输

#### 优化前

每次更新传输完整的卫星对象：

```json
{
  "noradId": "25544",           // ~10 B
  "name": "ISS (ZARYA)",        // ~15 B
  "position": { "lat": 45.2, "lng": 120.5, "alt": 420000 },  // ~50 B
  "timestamp": "2026-04-08...",  // ~25 B
  "countryCode": "CIS",          // ~5 B
  "mission": "Manned",           // ~20 B
  "operator": "NASA"             // ~15 B
}
```

**每颗卫星数据量**：~140 字节
**每次传输总量**：16,204 × 140 = 2.16 MB
**每分钟传输总量**：2.16 MB × 20 = 43.2 MB

#### 优化后

**初始化时一次性传输元数据**：

```json
{
  "25544": { "name": "ISS (ZARYA)", "countryCode": "CIS", "mission": "Manned", "operator": "NASA" }
}
```

**元数据总量**：16,204 × 65 = 1.03 MB（仅传输一次）

**每次更新只传输位置**：

```json
{ "noradId": "25544", "lat": 45.2, "lng": 120.5, "alt": 420000 }
```

**每颗卫星数据量**：~55 字节
**每次传输总量**：16,204 × 55 = 0.85 MB
**每分钟传输总量**：0.85 MB × 20 = 17.0 MB

#### 数据传输对比

| 项目 | 优化前 | 优化后 | 节省 |
|------|--------|--------|------|
| 每次传输 | 2.16 MB | 0.85 MB | 1.31 MB |
| 每分钟传输 | 43.2 MB | 17.0 MB | 26.2 MB |
| 每小时传输 | 2.59 GB | 1.02 GB | 1.57 GB |

---

### 2. 内存使用

| 阶段 | 优化前 | 优化后 | 变化 |
|------|--------|--------|------|
| 页面加载后 | 151.9 MB | 120.4 MB | -20.7% |
| 稳定运行后 | 192.3 MB | 163.7 MB | -14.9% |

**分析**：
- 内存减少主要来自 Worker 传输数据的结构简化
- 主线程不再重复存储不变的元数据字段

---

### 3. 帧率 (FPS)

| 测试 | 优化前 | 优化后 |
|------|--------|--------|
| 样本 1 | 61 fps | 55 fps |
| 样本 2 | 61 fps | 60 fps |
| 样本 3 | 49 fps | 56 fps |
| 样本 4 | 60 fps | 54 fps |
| 样本 5 | 60 fps | 61 fps |
| **平均** | **58.2 fps** | **57.2 fps** |

**分析**：
- FPS 略有下降，但仍在 55+ fps 范围内
- 可能是合并 metadata 和 positions 的 computed 计算开销
- 实际使用体验无明显差异

---

### 4. 加载性能

| 指标 | 优化前 | 优化后 | 变化 |
|------|--------|--------|------|
| LCP | 1,137 ms | 1,144 ms | +0.6% |
| TTFB | 6 ms | 7 ms | +1 ms |
| TLE API 耗时 | 1,471 ms | 1,296 ms | **-11.9%** |

**分析**：
- LCP 基本持平，由 Cesium 初始化主导
- TLE API 耗时减少，可能与网络波动有关

## 代码变更

### 修改文件

| 文件 | 变更内容 |
|------|----------|
| `orbit.worker.ts` | 新增 metadata 消息类型，positions 只返回位置数据 |
| `useOrbitWorker.ts` | 新增 metadata ref，处理新消息类型 |
| `useLocalSatellites.ts` | 在 computed 中合并 metadata 和 positions |

### 新增类型

```typescript
// orbit.worker.ts
export interface SatelliteMetadata {
  name: string
  countryCode?: string
  mission?: string
  operator?: string
}

export interface PositionData {
  noradId: string
  lat: number
  lng: number
  alt: number
}
```

## 结论

### 优化效果

| 方面 | 效果 |
|------|------|
| **数据传输** | ✅ 显著减少 60.6% |
| **内存使用** | ✅ 减少 14.9% |
| **FPS** | ⚠️ 略微下降 1.7%，但仍在可接受范围 |
| **加载性能** | ➖ 基本持平 |

### 建议

1. **正优化确认**：整体为正优化，数据传输和内存使用均有显著改善
2. **后续优化方向**：
   - 考虑使用 Web Worker 的 `Transferable` 对象进一步减少传输开销
   - 可对 computed 合并逻辑做性能分析，看是否有优化空间
   - 考虑按视野范围只传输可见卫星数据
